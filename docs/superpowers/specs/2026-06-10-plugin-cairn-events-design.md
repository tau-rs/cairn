# Plugin host slice 4: cairn events (host → plugin)

**Date:** 2026-06-10
**Status:** Design — approved, pre-implementation
**Builds on:** slices 1/3a/3b/3c (the plugin host + host-callbacks) and slice 2 (the SDK), [ADR-0008](../../decisions/0008-plugin-host.md)

## Goal

Let the host **push cairn change events to subscribed plugins** so a plugin can
react (e.g. regenerate a derived index when a note changes). This inverts the
existing direction (so far plugins are invoked and call *back* to the host); now
the host notifies plugins. It is delivered as a **request/response reusing the
existing dispatch loop**, so an event handler can make capability-gated
host-callbacks (read/write the cairn in reaction).

## Why request/response, not fire-and-forget

A plugin is a synchronous one-in-flight stdio server. A fire-and-forget
notification would deadlock the instant an event handler made a host-callback —
the host only reads a plugin's stdout *during* an active request. Delivering each
event as an ordinary `Request` (with an `id`) and running the **same dispatch loop
as `invokeCommand`** means: (a) no new "notification" concept — it rides the
existing `Request`/`Response` types; (b) handlers can read/write/delete/search in
reaction, capability-gated; (c) the host blocks until the plugin acks, consistent
with the one-in-flight model.

## Architecture & timing

```
engine Event (NoteChanged/NoteDeleted)
   │  (emitted to the EventSink during a command or watch-change)
   ▼
daemon: after the operation completes, for each note event   ── still broadcasts to WS as today
   ▼
Engine::dispatch_plugin_event(&PluginEvent, sink)   ── mem::replace host out (re-entrancy, like invoke)
   ▼
ProcessPluginHost::dispatch_event   ── for each plugin declaring the `events` capability:
   ▼
   write `cairn/event` Request → run the invoke dispatch loop (callbacks allowed) → plugin acks
```

**Timing is the safety guarantee.** The daemon forwards events **after** the
originating command/watch-change returns — so no plugin is mid-`invokeCommand`
when an event is delivered. Everything runs on the same thread under the daemon's
existing engine `Mutex`, so there is no concurrency with an in-flight invoke and no
interleaving on a plugin's stdio. Delivery is **synchronous**: a command/change
waits for subscribed plugins to handle the event before returning (the same
one-in-flight, no-timeout behavior the host already has; a hanging handler blocks
the daemon — unchanged limitation, async decoupling is a future slice).

**Non-recursive.** Events *generated by* an event handler (a plugin writes a note
on `noteChanged` → a new `NoteChanged`) still broadcast to WS but are **not**
re-forwarded to plugins in the same cycle — this prevents infinite loops.

## Decisions (resolved during brainstorming)

- **Term:** `cairn` (the project's established term for the note collection;
  replaces the leftover "vault"). Method `cairn/event`, type `CairnEvent`.
- **Subscription:** a single **`events` capability**. The host delivers *all* cairn
  events to any plugin that declares it; the plugin filters by kind itself
  (no per-kind manifest subscriptions — a later refinement).
- **Events delivered:** `NoteChanged` and `NoteDeleted` only. `Reindexed`/`Committed`
  are internal and not forwarded.
- **Daemon-only.** Plugins are loaded only by the daemon (the CLI uses
  `NoopPluginHost`), so forwarding wires only there.
- **Layering:** the `PluginHost` port (in `cairn-ports`) takes a wire-agnostic
  ports type `PluginEvent`; the infra host maps it to the protocol's wire
  `CairnEvent` (same pattern as `PluginInfo` ← `InitializeResult`). The engine and
  ports stay free of any `cairn-plugin-protocol` dependency.

## Components

### 1. Protocol — `crates/cairn-plugin-protocol/src/lib.rs`

```rust
/// Host -> plugin: a cairn change event. Delivered to plugins declaring `events`.
pub const METHOD_CAIRN_EVENT: &str = "cairn/event";

/// Capability: receive pushed cairn events.
pub const CAP_EVENTS: &str = "events";

/// The kind of a cairn change.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub enum CairnEventKind {
    NoteChanged,
    NoteDeleted,
}

/// Params of the `cairn/event` request. Ack result is an empty object `{}`.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct CairnEvent {
    pub kind: CairnEventKind,
    pub path: String,
}
```

`cairn/event` is a normal `Request` (carries an `id`); the plugin replies
`Response{id, result: {}}`. `#[serde(rename_all = "camelCase")]` → the kinds
serialize as `"noteChanged"` / `"noteDeleted"`.

### 2. Ports — `crates/cairn-ports/src/lib.rs`

A wire-agnostic event type + a delivery method on `PluginHost`:

```rust
/// A cairn change the host may push to subscribed plugins.
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum PluginEvent {
    NoteChanged(NotePath),
    NoteDeleted(NotePath),
}
```

`PluginHost` gains:

```rust
    /// Deliver a cairn event to every loaded plugin that declared the `events`
    /// capability, servicing any host-callbacks each makes while handling it.
    /// Best-effort: a plugin that errors or whose pipe breaks is logged + skipped.
    fn dispatch_event(&mut self, event: &PluginEvent, callbacks: &mut dyn PluginCallbacks);
```

`NoopPluginHost::dispatch_event` is a no-op. `PluginHost::invoke`'s existing
signature is unchanged. (`NotePath` is already imported in `cairn-ports`.)

### 3. Plugin host delivery — `crates/cairn-infra/src/plugin_host.rs`

`ProcessPluginHost::dispatch_event` iterates `self.loaded`; for each plugin whose
`capabilities` contains `CAP_EVENTS`, it maps the `PluginEvent` to the wire
`CairnEvent` and calls a new `LoadedPlugin::deliver_event(cairn_event, callbacks)`.

`deliver_event` mirrors `invoke_command`: build a `Request { method:
METHOD_CAIRN_EVENT, params: <CairnEvent> }`, then run the **same dispatch loop** —
read `Incoming`, service callback `Request`s (capability-gated, via the existing
`service_callback`), return when the matching-id `Response` arrives. The ack
result is ignored (the plugin returns `{}`). Per-plugin errors are logged to
stderr and do not abort the loop over other plugins (mirrors `load`'s
skip-and-log). `PluginEvent::NoteChanged(p)` → `CairnEvent { kind: NoteChanged,
path: p.as_str().to_string() }`, likewise `NoteDeleted`.

(`invoke_command`'s loop body — read `Incoming`, `Response`→return,
`Request`→`service_callback`+write — should be factored into a shared helper that
both `invoke_command` and `deliver_event` call, to avoid duplicating the loop.)

### 4. Engine — `crates/cairn-app/src/lib.rs`

```rust
/// Deliver a cairn event to subscribed plugins (best-effort). Event-handler
/// callbacks route through the engine and their events emit to `sink`.
pub fn dispatch_plugin_event(&mut self, event: &PluginEvent, sink: &mut dyn EventSink) {
    let mut host = std::mem::replace(&mut self.plugins, Box::new(NoopPluginHost));
    {
        let mut cb = EngineCallbacks { engine: self, sink };
        host.dispatch_event(event, &mut cb);
    }
    self.plugins = host;
}
```

Same `mem::replace` re-entrancy as `invoke_plugin_command`. Returns nothing
(best-effort). The public signature is new; only the daemon calls it.

### 5. Daemon wiring — `crates/cairn-daemon/src/lib.rs`

Today `run_command_blocking`/`apply_change_blocking` pass a `BroadcastSink` to the
engine, which emits straight to the broadcast channel. To *also* forward note
events to plugins, replace the sink with an **`EventTap`** that broadcasts **and**
collects:

```rust
struct EventTap {
    tx: broadcast::Sender<WireEvent>,
    collected: Vec<AppEvent>,
}
impl EventSink for EventTap {
    fn emit(&mut self, e: AppEvent) {
        self.collected.push(e.clone());
        let _ = self.tx.send(app_event_to_wire(e));
    }
}
```

After the engine operation returns, forward each collected `NoteChanged`/
`NoteDeleted` to plugins (a fresh `BroadcastSink` for handler-generated events —
broadcast-only, so they are **not** re-forwarded → non-recursive):

```rust
pub fn run_command_blocking(&self, command: &Command) -> Result<CommandResponse, ServiceError> {
    let mut guard = self.engine.lock().expect("engine mutex poisoned");
    let mut tap = EventTap { tx: self.events.clone(), collected: Vec::new() };
    let result = dispatch_command(&mut guard, command, &mut tap);
    let events = tap.collected; // take ownership; tap (its &tx clone) dropped
    if result.is_ok() {
        for pe in events.iter().filter_map(to_plugin_event) {
            let mut fwd = BroadcastSink(self.events.clone());
            guard.dispatch_plugin_event(&pe, &mut fwd);
        }
    }
    result
}
```

`apply_change_blocking` gets the same treatment (collect via `EventTap`, then
forward). A free `fn to_plugin_event(e: &AppEvent) -> Option<PluginEvent>` maps
`NoteChanged`/`NoteDeleted` and returns `None` for `Reindexed`/`Committed`. The WS
path is unchanged (events still broadcast on `emit`).

### 6. SDK — `crates/cairn-plugin-sdk/src/lib.rs`

```rust
impl Plugin {
    /// Register a handler for pushed cairn events. Called with capability-gated
    /// `Host` access (the handler may read/write the cairn in reaction).
    pub fn on_event<F>(&mut self, handler: F)
    where
        F: FnMut(CairnEvent, &mut Host<'_>) -> Result<(), PluginError> + 'static;
}
```

The run loop, on a `cairn/event` request, deserializes `CairnEvent`, calls the
registered handler (errors → the response carries the error; the host logs it),
and replies `{}`. If no handler is registered, it acks `{}` (harmless). `CairnEvent`
is re-exported from the SDK.

### 7. Example plugin — `crates/cairn-plugin-example`

Declare `capabilities = ["events", "fs:write"]` (in the event tests' manifests).
Register an event handler that, on `NoteChanged`, writes a marker note via
`host.write_note(...)` — proving event delivery **and** callback-during-event
**and** dual-capability gating. (The existing command set stays.)

`MapCallbacks` already supports the callbacks an event handler uses. The daemon
forwarding gets a focused test (an `EventTap`-collected command produces a
`PluginEvent` forwarded to a stub host) if practical, or is covered by the e2e
delivery path.

## Out of scope

- Async/decoupled event delivery (the current synchronous-under-Mutex model is the
  walking skeleton); per-kind manifest subscriptions; `Reindexed`/`Committed`
  delivery; the `net`/`agent` capabilities; UI plugins (the UI session's).
- A delivery timeout (same no-timeout limitation as `invoke`; tracked as the
  prioritized hardening across all plugin slices).

## Unchanged

`cairn-contract`, `cairn-cli`, `Engine::invoke_plugin_command`'s signature, the
`PluginHost::invoke` signature, and the WS broadcast path (events still flow to WS
exactly as today; plugin forwarding is additive).
