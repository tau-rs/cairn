## 1. What Cairn is (one paragraph)

Cairn is an open-source, git-backed, Obsidian-class note app. The Rust **engine** (`cairn-core`, this repo) is a transport-blind hexagon: one async contract (commands / queries / events) is consumed by a CLI today and by your **web UI** next. It targets desktop (Tauri), mobile (Tauri v2), pure browser, and remote â€” all served from the *same* core via in-process (first) or daemon (later) transports. Storage is plain markdown files under git; a CRDT live-collab layer is a designed seam. It is built for tau (Titouan's agent runtime) integration. Lives in the `tau-rs` org. A single note collection is called **"a cairn"** (the only renamed term; everything else is standard: note, link, backlink, tag, plugin, index, graph).

## 2. Current engine state (what actually works)

A **walking skeleton**, fully tested (28 tests, clippy/fmt clean, CI in place):

- `cairn-domain` â€” pure model: `NotePath`, `Note` (frontmatter + body), `[[wikilink]]` extraction, backlink `Graph`.
- `cairn-ports` â€” trait ports: `VaultStore`, `SearchIndex`, `Vcs`, plus seam traits `Watcher`, `Executor`, `CollabSession`, `AgentRuntime`.
- `cairn-infra` â€” real adapters: `LocalFsStore` (filesystem), `GitVcs` (git2), `InMemoryIndex` (substring search); seam adapters `NoopWatcher`, `BlockingExecutor`, `NoCollab`, `NullRuntime`.
- `cairn-app` â€” `Engine<S,I,V>` use-cases: write/read/delete note, search, backlinks, commit, reindex â€” emitting `Event`s via an `EventSink`.
- `cairn-contract` â€” the wire contract DTOs + **generated TypeScript bindings** (your import surface).
- `cairn-cli` â€” the `cairn` binary, an in-process consumer (`init / write / read / search / backlinks / commit`).

Build & run:

## 3. The contract you build against

Generated TypeScript lives at **`crates/cairn-contract/bindings/`** (`Command.ts`, `Query.ts`, `Event.ts`) â€” committed, regenerated by `cargo test -p cairn-contract`. Import these directly; they are the source of truth and match the Rust serde wire format exactly (tagged unions, discriminant field `"type"`, snake_case tags).

```ts
// Command.ts
type Command =
  | { type: "write_note";  path: string; contents: string }
  | { type: "delete_note"; path: string }
  | { type: "commit";      message: string };

// Query.ts
type Query =
  | { type: "get_note";      path: string }
  | { type: "search";        query: string }
  | { type: "get_backlinks"; path: string };

// Event.ts  (push, server -> client)
type Event =
  | { type: "note_changed"; path: string }
  | { type: "note_deleted"; path: string }
  | { type: "committed";    commit: string }
  | { type: "reindexed";    count: number };
```

**Design contract for the client:** treat the engine as **async requestâ†’response + an event stream**, and NEVER assume it's in the same process. Write the UI against a small transport-abstracted client interface (e.g. `sendCommand(c): Promise<â€¦>`, `runQuery(q): Promise<â€¦>`, `subscribe(cb: (e: Event) => void)`) with a swappable implementation. That single discipline lets the same UI run over Tauri IPC (in-process) and the daemon (network) unchanged.

## 4. âš ď¸Ź The one real gap to know about

The contract types exist and are validated (serde round-trip + TS codegen), **but nothing serves them yet.** The CLI calls `Engine` methods directly; there is no dispatcher mapping `Command`/`Query` â†’ `Engine` calls, and no mapper from the app's internal `Event` to the contract `Event` (note: app uses `count: usize`, contract uses `count: number`/`u32` â€” a narrowing the mapper must handle). This was intentional for the skeleton.

**Therefore the first integration task (opening task of the transport/daemon sub-project) is:**
1. A `dispatch(Command) -> Result` and `query(Query) -> Result` over an `Engine`.
2. `From<cairn_app::Event> for cairn_contract::Event`.
3. A query-response shape (the `Query` variants currently have no defined response DTOs â€” e.g. `get_note` â†’ note contents, `search` â†’ list of paths, `get_backlinks` â†’ list of paths â€” these response types still need to be added to `cairn-contract`).
4. The chosen transport: **in-process/Tauri first** (recommended), daemon later. Codegen tooling can stay `ts-rs`, or move to `tauri-specta`/`rspc` if you want command wiring generated too.

Until that exists, you can build the UI against the committed TS types with a mock client, then swap to the real transport when the dispatcher lands. Coordinate with Titouan on whether the UI session also builds the dispatcher+Tauri shell or whether that's a preceding engine task.

## 5. Architecture you're plugging into (so your choices fit)

- **Primary shell:** Tauri (desktop + mobile), core in-process â€” most efficient, Obsidian-like, offline-first.
- **Secondary:** a daemon (HTTP/WS) for pure-browser + remote/multi-device â€” same contract, network transport.
- **Engine threading is never in the browser:** the browser is served by the daemon where the engine runs native; you render results + consume events.
- **Plugins (future):** one manifest, an optional engine part (tau-style, out-of-process) and an optional **UI part (JS/TS in the webview)** â€” your session will eventually host the UI-plugin API; the engine-side extension points (commands, vault events, content processors, import/export) are specified in the design doc Â§7.
- **Tau (future):** agent actions on notes, cairn-as-agent-tool, agent threads stored in git â€” exposed via the `AgentRuntime` port (currently `NullRuntime`). Tau is still in active build; keep this loosely coupled.
